# Реализация по требованиям

## Предметная область

Проект Meeple Scope решает задачу подбора настольных игр под компанию: каталог, фильтры по игрокам и времени, аналитика, сценарии вечера, обучение правилам, рейтинг игроков, друзья и лобби.

## Стек

| Требование            | Реализация                                                                                                                   |
| --------------------- | ---------------------------------------------------------------------------------------------------------------------------- |
| React 18 + TypeScript | `src/main.tsx`, функциональные компоненты, типы в `src/entities/**/model`                                                    |
| TanStack Router       | маршруты в `src/app/router.tsx`                                                                                              |
| State manager         | `Zustand` в `src/features/finder/model/preferencesStore.ts`; выбран для компактного состояния подбора без лишней шаблонности |
| CSS Modules           | стили страниц, layout и UI-компонентов в `*.module.css`                                                                      |
| Fetch API             | общий клиент `src/shared/api/http.ts`, компоненты не вызывают `fetch` напрямую                                               |
| ESLint + Prettier     | скрипты `npm run lint`, `npm run format`                                                                                     |

## Качество кода

В проекте используются кастомные хуки для инкапсуляции логики:

- `useAuth` — авторизация, регистрация, выход и получение текущего пользователя.
- `usePlayers` — загрузка игроков, поиск по идентификатору, заявки в друзья.
- `useLobby` — получение лобби, приглашение друзей и добавление гостей.
- `useBoardGameData` — каталог игр, карточка игры, статистика, категории, механики и темы вечеров.
- `useProfile` — загрузка и сохранение профиля пользователя.
- `useReviews` — CRUD отзывов.
- `useAsyncResource` — общий хук для состояний загрузки, ошибок и повторной загрузки данных.

ESLint и Prettier настроены в `package.json`:

- `npm run lint` запускает проверку ESLint.
- `npm run format` запускает форматирование Prettier.

Код проекта проверен командой `npm run lint` и проходит линтер без ошибок.

## API и хранение данных

Рабочая база данных: PostgreSQL. Схема находится в `server/schema.sql`, миграция в `server/migrate.mjs`, импорт данных в `server/seed.mjs`, REST API в `server/index.mjs`.

Основные таблицы:

- `games`, `categories`, `mechanics`, `progression_paths`, `game_night_themes`, `tools` — каталог и справочники настольных игр.
- `profiles`, `reviews` — профиль и CRUD отзывов.
- `app_users`, `auth_sessions` — регистрация, вход и серверные сессии.
- `friend_requests`, `friendships` — заявки в друзья и подтвержденная дружба.
- `lobbies`, `lobby_members` — лобби пользователя, приглашенные друзья и гости.

JSON-файл с базой не используется. `server/seed.mjs` читает исходный CSV из `data/bgg_dataset.csv` и импортирует данные напрямую в PostgreSQL. Приложение читает рабочие данные только через Express API на `http://localhost:3001`.

## CRUD и REST

| Ресурс      | Endpoints                                                                      | Где используется                              |
| ----------- | ------------------------------------------------------------------------------ | --------------------------------------------- |

## Маршруты

В приложении больше трех осмысленных маршрутов:

- `/` — дашборд с аналитикой и трендами.
- `/catalog` — каталог и фильтры.
- `/finder` — подбор игры.
- `/players` — рейтинг, поиск друзей, входящие и исходящие заявки.
- `/lobby` — сбор состава вечера.
- `/profile` — профиль и CRUD отзывов.
- `/auth` — вход и регистрация.
- `/learn/$slug` — обучение и генератор сценария для конкретной игры.

## Авторизация и друзья

Регистрация принимает почту и пароль, создает запись в `app_users`, автоматически формирует внутренний `username`, хэширует пароль и выдает токен из `auth_sessions`. Токен хранится во фронтенде в `localStorage` и автоматически добавляется в заголовок `Authorization: Bearer ...` через `localRequest`.

Дружба не выставляется флагом. Пользователь ищет другого игрока по 10-значному `friendCode`, который генерируется случайно при регистрации, отправляет заявку через `POST /friend-requests`, а принять или отклонить ее может только получатель через `PATCH /friend-requests/:id`. Только подтвержденные друзья появляются в лобби как приглашенные игроки.

## Лобби

Лобби привязано к текущему аккаунту через `owner_user_id`. В него можно добавить:

- принятого друга из `friendships`;
- гостя вручную, если у человека нет аккаунта.

Сервер проверяет, что приглашаемый пользователь действительно является другом владельца лобби.

## Каталог

Каталог выводит все игры из PostgreSQL через серверную пагинацию: `GET /games?_page=N&_per_page=50&_sort=name`. На странице отображается 50 игр, сортировка выполняется на сервере по имени, снизу есть кнопки и выбор номера страницы.

Фильтры по названию, числу игроков, максимальной длительности и режиму каталога передаются в API, поэтому пользователь не ограничен первой порцией данных.

## Подбор, обучение и генератор

Подбор поддерживает варианты без ограничений по числу игроков и длительности, а сценарий вечера можно оставить `Любой`.

Обучение и генератор сценариев больше не используют статичный шаблон. Страница игры вызывает backend endpoint `POST /ai/learning-plan`. Сервер берёт:

- карточку игры из PostgreSQL;
- описание, механики и категории из BoardGameGeek XML API2, если у игры есть `bgg_id` и доступен `BGG_API_TOKEN`;
- выбранную цель объяснения и тему вечера;
- локальную модель Ollama, по умолчанию `qwen2.5:1.5b`.

Если BGG XML API2 недоступен, сервер использует локальную PostgreSQL-базу и просит модель не придумывать официальные правила. Если Ollama недоступен, пользователь видит ошибку с инструкцией по запуску модели. Статичная подмена результата не используется.

## Архитектура

Проект разделен на слои:

- `app` — роутер и layout.
- `entities` — типы, API-сервисы и хуки доменных сущностей.
- `features` — пользовательское состояние и сложная логика подбора.
- `pages` — страницы маршрутов.
- `shared` — общий API-клиент, утилиты и UI.
- `server` — Express API, схема PostgreSQL и сидирование.

Компоненты страниц используют хуки и сервисы, а не прямые HTTP-запросы. Состояния загрузки, ошибок и пустых списков закрываются компонентом `StatusBox`.

## Запуск

База поднимается через Docker:

```bash
docker compose up -d
npm run db:migrate
npm run db:seed
```

После этого:

```bash
npm run api
npm run dev
```
